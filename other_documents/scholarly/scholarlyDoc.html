1. Introduction

1.1 Purpose

This SRS specifies the requirements for the Wind Turbine Web Manager app, v1.0. The product's scope consists of a web application, an API and also a RSS feed that delivers information available in the web app.

1.2 Document Conventions

  • The document is based on the template IEEE System Requirements Specification Template .
  • This document repects the Scholarly HTML specification
  • Priorities for higher-level requirements are assumed to be inherited by detailed requirements.
  • Citations are specified in italics.
  • References to the elements of a web page use a code-like font.
  • Priorities for functionalities are higlighted in bold.

1.3 Intended Audience and Reading Suggestions

This document is intended for the development team and for the teachers that are going to be grading the product and documentation. For the first review that is going to take place in the 8th week, the relevant sections are section 3. External Interface Requirements and 4. System Features.

1.4 Product Scope

The Wind Turbine Web Manager is intended to help managers and business owners to manage their owned wind turbines by letting them visualize the state of a turbine at any moment through the web app, transfer wind turbines between users, notify the owner when in need of maintenance, or obtain formated data through an API or RSS feed.

2. Overall Description

2.1 Product Perspective

This product originates from one of the project proposals for the Web Technologies course, class 2A6, a project of category M. It is a new, self-contained product which contains the following subsystems:

2.2 Product Functions

  • Account features
    • Users can manage an account inside the app.
  • Card view features
    • Users can see restricted info about public wind turbines owned by other users.
    • Users can see detailed info about turbines they own.
  • General stats features
    • Users can see statistics about all their turbines as a total.
  • Detailed view features
    • Users can see detailed info about each turbine.
    • Users can change the visibility of a turbine.
    • Users can create new turbines in a suitable location.
  • Turbine creation features
    • Users can create new turbines in a suitable location.
  • Filtering Features
    • Users can filter through public or private turbines.
    • Users can search public or private turbines by name.
  • Transfer features
    • Users can make transfers of wind turbines between accounts.
  • Notifications features
    • The system should notify users if a turbine experiences an abnormal event or if they receive a request for a transfer.

2.3 Uses Classes and Characteristics

The sole users of this application are managers that deal with a business's resources in terms of energy production. Through the application they can monitor the state of the turbines, they can stop them if needed and they can monitor the efficiency and output of the stations

2.4 Operating Environment

The app's v1.0 version should run as expected on any of the newer secure versions of modern browsers (Firefox 88-98, Chrome 93-99, Opera 72-83, Safari 11.1.1-15.2).

2.5 Design and Implementation Constraints

The main constraints are the fact that according to grading policy, the developers aren't allowed to use frontend or backend frameworks, and all other tools need to have an open license. As such, for the development of the interface we are restricted to HTML5, CSS3 and Javascript, and for the backend we'll also use Javascript through node.js, but without the use of a framework for web servers like express.

2.6 User Documentation

A relevant guide to using the app will be available for users through the landing page of the web app.

2.7 Assumptions and Dependencies

The Wind Turbine Web Manager relies on a series of APIs like Google Maps API and OpenWeather API to provide relevant data to our virtual Wind Turbines, to help when deciding the suitability of a new place for placing a wind turbine, and to illustrate data, so any changes to these API's specifications will inevitably lead to changes in the application. The development team will also rely on libraries like Chart.js for displaying data, so the app might also be in need of an update as dependencies update.

3. External Interface Requirements

3.1 User Interface

A user can interact in the Graphical User Interface with the following pages:

  • Landing Page
  • Login Page Interface
  • Sign Up Page Interface

    • After logging in:

  • Account Info Page
  • Public Turbines Page
  • Owned Turbines Page
  • Turbine Details Page
  • Add Turbine Page

The dominant theme and color in the app's interface is the sky's blue and white color scheme.

3.1.1 Landing Page

On the landing page there are 2 buttons, the Login one leading to the login page and the Get started one to a guide on how to use our app.

3.1.2 Login Page Intreface

On the login page there is a window, containing the field which must be filled with the registered user's info before pressing the Log in button, which will lead to their owned turbines page. The Forgot password button leads the user to the reset password page. The Register button leads the user to the register page if the user wants to create a new account.

3.1.3 Sign Up Page Interface

On the register page there is a window, containing the field which must be filled with the registered user's info before pressing the Sign Up button, which will lead to their owned turbines page. The Forgot password button leads the user to the reset password page. The Login button leads the user to the log in page if the user wants to log in with an existing account.

3.1.4 Account Info Page

In the account info page, a user is able to change any of it's account data as long as it is still valid, and by pressing Done he is taken to the owned turbines page.

3.1.5 Public Turbines Page

Interacting with the navbar above the page an user can go to the page containing the turbines they own by accessing Owned, they can see the turbines whose info is public by pressing Public, through pressing the bell icon they are redirected to a page with their latest notifications, and by pressing the 3 line menu they have access to the logout button which takes them back to the landing page, and by pressing Account they are redirected to the account info page.

Under the navbar in the filter section, they can use the search bar to look for a turbine with a specific name, or use the filters to to only see turbines that conform to the filter. Under the filter section there are rows of cards containing data about each public wind turbine(their name, location and their wear as a color inside the spinner in the top left of the card). By clicking one of the cards the user gets to see an overlay with more detailed information(picture, contact data for the owner, stats about the wind turbines, the wear of the turbine over time).

3.1.6 Owned Turbines Page

At the top of the page there is the same navbar as in the Public Turbines Page. Under the navbar in the filter section, they can use the search bar to look for a turbine with a specific name, or use the filters to only see turbines that conform to the filter.

By pressing the + button, the user is redirected to the add turbine page. In the remaining space, there are cards that contain detailed info about each turbine, an image of the turbine and a graph of its wear over time. By pressing the Details button on any of the pages, the user accesses the turbine details page for that turbine.

3.1.7 Turbine Details Page

At the top of the page there is the same navbar as in the Public Turbines Page. Below the navbar the user can see a card, with data about himself, a picture of the turbine, and the year of construction and the turbines total energy production. Below the user can find 3 buttons, Stop stops the turbine until pressed again, Delete removes the turbine from the user's account, Set private sets the turbine as invisible to the public.(Start if the turbine is stoppde and Set public if the turbine is private).

On the left hand side of the page there is a navbar with 5 sections, each of them showing different information below the existent card:

  • State brings up a chart showing the wear over time.
  • Wind shows a graph with the changes in wind speed over time.
  • Energy illustrates a progression with the amount of energy produced.
  • Efficiency illustrates the changes in efficiency of the turbine.

3.1.8 Add Turbine Page

On this page the user can add a new wind turbine to his account, by choosing a location on the map on the left side, for which he will get data about the location's coordinates and the suitability of placing a turbine there. On the left side he can fill up the data of the turbine that isn't autofilled and press Create to add it to his account.

3.1.9 Reset Password Page

On this page, the user will fill the Email field withe the mail address for which they wish to reset the password. When the Done button is clicked, and email containing a new temporary password which can be used to log into the account. The user can later change their password in the account details page.

3.1.10 Notifications Page

On this page, a user can manage the notifications they receive about purchase requests from other users and alerts about their owned turbines. Pressing the Accept button will transfer the requested turbine to the buyer and the Decline button will refuze the offer. In an alert tab, pressing the Check it out button will redirect the to the user turbine details page of the turbine for which the alert was generated. Pressing the X button witl ignore the alert.

3.2 Hardware Interfaces

The web app can be used on all devices which support a modern GUI web browser (Google Chrome, Mozilla Firefox, Safari, Edge, Opera, etc), like computers, laptops, mobile devices(Tablets, smart phones), and have a screen width bigger than Samsung Edge, as the the design of the app is responsive and can adapt to different screen sizes. For the API the HTTP protocol is used and data is sent and received in json format.

3.3 Software Interfaces

To obtain data related to users, for example in the case of authentication, our application interacts with a MongoDB database through javascript. To obtain data related to information about the wind turbines, the application makes GET requests to our API which returns formatted data based on the request. For creating a new turbine a POST request is used. For the user interface Chart.js is used to display charts, and Google Maps API to display location info. For sending notifications the Notification API will be used. Javascript is also used to maintain responsive design as a means of improving user experience.

3.4 Communications Interfaces

The web app requires an internet connection to be used. For communication with the API the app uses the HTTP protocol, and for sending notification emails to users we'll use the nodemailer node.js package, which uses the SMTP protocol.

4. System Features

4.1 Account Management

This high priority feature provides users the capability to login if they have an account or sign up to the web application by providing personal data, and manage their info through the account info page.

Functional Requirements

In the login page, if the user inputs an email and password that don't belong to any registered account and tries to log in, he will be informed with a red warning text below the fields that either the email or password is wrong.

In the Sign Up Page, after the Done button has been pressed, for fields First name, Name and Company the server will verify that they have a certain length, for Mail and Phone Number verify that a valid phone number and email address has been filled. The CNP field can be validated according to the CNP validation formula, Adresa is a select element from which the user can select their county, and Data Nastere a input element of type date, both of them validated by the CNP. For the Password a strength analysis should be implemented. If any of the form elements are invalid, the sign up fails and the user is informed with warning text which fields failed the verification.

In the account info page, the same verifications are done for any of the fields changed as in the sign up page. To change the current password the user needs to input the old and the new password.

4.2 Card View

This high priority feature allows the user to see limited info about public turbines and owned turbines in the from of cards with varying level of detail, depending on if a turbine is owned by the user or is visible publicly.

Functional Requirements

In the public turbines page, a user will see a grid of cards, each containing the name, wear, location and company of a public wind turbine. If there are no such turbines, in the middle of the screen will be displayed the message “There are no public turbines available for viewing”. By clicking on a card, an overlay is displayed, showing the name, year of construction, and average energy production of the turbine, the company, email and phone number of the owner, an image of the turbine and a chart showing the wear over time.

On the owned turbines page, the cards displayed by default have the same level of detail as the cards on the public turbines page after they have been clicked.

4.3 Detailed View

This high priority feature let's users see a dedicated page with detailed info about a single one of their owned turbines.

Functional Requirements

In the turbine details page, the user has the ability to stop a turbine from working(for repairs), to remove the turbine from the account, or to set its visibility as public or private. He should also be able to see the location of the turbine on a map, and have access to charts detailing its energy production, efficiency in terms of potential output to actual output, the variance of wind speed and the progression of wear on the turbine.

4.4 Transfer

This medium priority feature should allow the transfer of public turbines between users.

Functional Requirements

By pressing Purchase inside a detailed view in the public turbines page, the owner of the turbine for which a transfer request was made should be notified of the request, and he can press Accept in the Notification menu to accept a request, or press Remove to delete the notification. If the request is accepted, the requester will find the transferred turbine in his account, and it will be removed from the user that accepted the request. If the turbine no longer exists in the account of the user that received the request, he will be notified that he can't transfer a turbine that he doesn't own. If the request is refused, the requester will also be notified that he was refused.

4.5 Filter and Search

The filtering feature has a low priority and it should allow users to filter through turbines based on varying criteria.

Functional Requirements

In the public turbines page and owned turbines page, a user should be able to filter through the cards displayed. Using the search bar the user engages in a filtering based on turbine name, and as each letter is typed in the search bar, in the grid will reamain only cards for the turbines whose name starts with the prefix typed in the search bar. If the search bar is empty all cards will be displayed. Through a several select menus, a user can filter through turbines by county, company, wear, and with two number input elements the user can filter by year of construction, selecting the turbines which have been built between two different years.

4.6 Notifications

Medium priority feature, users should be notified in the browser and through email about events on their account.

Functional Requirements

The system will send notifications to users in the following cases:

  • a turbine they own is worn out and in need of repairs
  • the wind speed exceeds a certain limit and a turbine is in danger
  • another user requested a transfer for one of the current user's turbines

The notifications should be sent through email and browser notifications, and the user should be able to see them by pressing the bell icon from the navbar. From there, if a notification refers to extreme conditions on a turbine they own, by clicking the notification the user will be taken to the details page of that turbine, if the notification is a request, the user can accept or deny it, and if it is a refusal they can just remove it.

4.7 General Stats

Low priority feature, the system offers visualization for data on all turbines a user owns as an ensemble.

Functional Requirements

In the owned turbines page, through the Statistici drop down menu, a user should be able to see a visualization of the average amount of energy produced, and one of the average efficiency of all of his turbines. Through the Harta drop down, a user should be able to see a map with the location of all of his turbines.

4.8 Turbine Creation

High priority feature, the system must create an interface for adding a new turbine to the users account.

Functional Requirements

The user should be able to pick a location on a map, and get the latitude and longitude of the location, the type of terrain and average wind speeds. To more easily inform the user about suitability, the location should also have a computed score about how suitable that location is for placing a turbine, 0 being least optimal and 1 optimal, this score indicated through a slider of different colors, red, yellow, green. The user is also able to assign a name and an image to the turbine he is about to add.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

No performance requirements specified.

5.2 Safety Requirements

Because in the case of the public turbines page, the contact info for the owner of each public turbine is available, we advise the owners of the accounts to create a dedicated email address for the web application.

5.3 Security Requirements

The app should employ prevention techniques for attacks like Cross Site Scripting and SQL Injection. The app can only store hashed versions of the user's passwords.

5.4 Software Quality Attributes

The application's architecture should follow the MVC pattern, so developers must keep track of separation of concerns. The code should have a focus on flexibility and be open to new features, which is always an option depending on user feedback. The development team must also make sure that the code is unit tested.

5.5 Buisness Rules

No business Rules imposed.

Other Requirements

The app should use the MVC architecture and be implemented through micro-services. The import and export of data should use open data formats.The database used can be relational or nosql.
